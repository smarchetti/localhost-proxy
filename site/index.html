Quick start

One-time DNS setup for the .test domain (writes /etc/resolver/test, needs sudo; delete the file to undo) 
lhp setup
Wrap your dev command in package.json 
"dev": "lhp -- next dev"
# servers that ignore PORT take a placeholder:
"dev": "lhp -- vite --port {port}"
Run dev in any worktree — it prints its stable URL and appears on the dashboard at http://localhost/

  │ worktree   feature-auth.tpg-code  (feature/auth)
  │ proxied    http://feature-auth.tpg-code.test

How it works

lhp wraps your dev command. It detects the worktree and repo from git (all worktrees share the main repository's .git dir), allocates a free port, exports it as PORT, registers name → port with a proxy daemon that auto-starts on first use, and unregisters when your command exits. Crashed processes are pruned automatically.

The daemon routes by hostname on port 80 (no root needed on modern macOS), passes WebSockets through so HMR works, and answers DNS for *.test from a built-in loopback-only responder — that's what lhp setup points macOS at.

Zero config, zero deps

Names come from your git layout. One self-contained binary pair, no runtime dependencies.

Repo-scoped names

feature-auth.my-repo.test and feature-auth.other-repo.test never collide. The dashboard groups by repo.

HMR just works

WebSocket upgrades are piped straight through to the dev server.

Self-healing routes

Routes persist across daemon restarts and vanish when their process dies.

Auth-flow safe

Auth.js/NextAuth env is injected so signin forms and OAuth callbacks stay on the proxy origin.

Friendly errors

Unknown or sleeping worktrees get a page listing everything that is running.

Configuration

Defaults: proxy on port 80, domain .test, scheme worktree.repo. Overrides live in ~/.lhp/config.json via lhp config; after changing them run lhp stop and the daemon restarts with the new settings on your next dev run.

CommandWhat it does
lhp -- <cmd…>Wrap a dev command — register, run, unregister on exit
lhp listRegistered worktrees and their URLs
lhp status / lhp stopDaemon status / stop the daemon
lhp config port 7777Move the proxy off port 80 (URLs gain :7777)
lhp config domain localhostZero-setup mode — *.localhost resolves natively, no sudo step
lhp config scheme worktreeFlat names without the repo subdomain
lhp setupOne-time /etc/resolver entry any non-localhost domain needs

Domains that can't work are refused with an explanation: .dev and .app are HSTS-preloaded (browsers force HTTPS), .local belongs to Bonjour. .test is IETF-reserved; anything made up works too.

Telling your app its public URL

Apps that build absolute URLs — auth flows, OAuth callbacks, share links — must build them on the proxy origin, or cookies and CSRF break the moment a form posts to localhost:<port>. Forwarded headers aren't enough in dev (the Next dev server rewrites them), so lhp injects environment variables instead:

LayerVariables
AlwaysPORT, LHP_URL, LHP_NAME
Auth.js defaultsAUTH_URL, NEXTAUTH_URL, AUTH_TRUST_HOST — only when you haven't set them; inert otherwise
Project configAnything your stack reads, via an "lhp" key in package.json
// package.json — {url}, {port}, {name} placeholders
"lhp": {
  "env": {
    "VITE_PUBLIC_URL": "{url}",
    "APP_HOST": "{url}",
    "AUTH_URL": null  // opt out of a built-in default
  }
}

Your shell environment always wins — an explicitly exported variable is never overridden. For OAuth providers (Okta, Google, …), whitelist each worktree's callback URL, e.g. http://feature-auth.my-repo.test/api/auth/callback/okta.
